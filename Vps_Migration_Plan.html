2. Pekerjaan Security Wajib Sebelum VPS Dibuka

P0 — Perbaiki keamanan Docker dan secret

Saat ini docker-compose.yml masih memiliki password database spesifik yang ditulis langsung:

POSTGRES_PASSWORD: Rooma_RasaEnak221372

Walaupun mungkin hanya password development, ini harus diperlakukan sebagai credential yang sudah terekspos karena berada di repository.

Yang harus dilakukan:
  • Ganti nilai Docker Compose menjadi environment variable, misalnya ${POSTGRES_PASSWORD}.
  • Jangan memakai password tersebut di VPS.
  • Jika password itu pernah dipakai di server/database nyata, lakukan rotasi.
  • Tambahkan .env.production.example atau dokumentasi daftar variable tanpa nilainya.
  • Pastikan file environment VPS memiliki permission ketat, misalnya hanya dapat dibaca user deployment.

P0 — Tambahkan .dockerignore

Saat ini .dockerignore belum ada, sementara root project mempunyai file .env.

Dockerfile menjalankan:

COPY . .

Tanpa .dockerignore, file .env, .git, log, dan file lokal lainnya dapat ikut masuk ke Docker build context. Walaupun tidak semuanya masuk final image, secret tetap berisiko masuk build cache atau layer builder.

Minimal .dockerignore harus mengecualikan: 
.env
.env.*
!.env.example
.git
.next
node_modules
npm-debug.log*
coverage

P0 — Perbaiki Dockerfile production

Dockerfile saat ini berisiko gagal pada build bersih karena:

  • npm ci dijalankan setelah hanya menyalin package.json dan package-lock.json.
  • package.json mempunyai script postinstall: prisma generate.
  • Pada tahap itu prisma/schema.prisma belum disalin.

Urutannya perlu diperbaiki dengan menyalin konfigurasi Prisma sebelum npm ci, atau mengatur proses generate secara eksplisit.

Selain itu, Docker Compose sekarang hanya menjalankan PostgreSQL. Belum ada service aplikasi Next.js production, health check aplikasi, konfigurasi network internal, atau prosedur migration.

3. Implementasi A4 dan A5

A4 — Rate limiting

Tambahkan rate limiting pada: POST /api/public/reservations

Rekomendasi untuk VPS:
  • Gunakan Redis untuk penyimpanan counter.
  • Key awal dapat menggunakan IP.
  • Setelah validasi minimal, bisa dikombinasikan dengan nomor telepon.
  • Pastikan request yang ditolak tidak membuat reservation maupun payment.
  • Kembalikan envelope aman dengan HTTP 429.
  • Tambahkan header seperti Retry-After.
  • Tambahkan automated test yang membuktikan request setelah batas tertentu menghasilkan 429.

Jika VPS hanya satu instance, pembatasan tambahan juga dapat diterapkan di Nginx. Namun application-level rate limit tetap diperlukan agar aturan bisnis tidak hanya bergantung pada reverse proxy.

A5 — Payload size limit

Endpoint saat ini langsung memanggil: await req.json()

Yang perlu ditambahkan:
  • Periksa Content-Length jika tersedia.
  • Tolak payload di atas batas dengan HTTP 413.
  • Tangani header yang tidak ada atau tidak valid.
  • Tetap jalankan Zod setelah payload lolos batas.
  • Atur batas tambahan di reverse proxy, misalnya client_max_body_size.
  • Tambahkan test untuk 400, 413, dan payload normal 201.
