Fase 1 — Blocking untuk Dev B
Dev B tidak bisa menyentuh Payment API sama sekali sebelum ini ada
01
Public reservasi API (tanpa login)
Endpoint utama yang menerima input dari tamu: pilih tanggal, sesi, jumlah orang. Ini adalah entry point seluruh alur bisnis.
entry point blocking dev B
02
Validasi capacity & blocked dates
Cek apakah sesi masih punya kapasitas dan tanggal tidak diblokir. Wajib ada sebelum reservasi bisa diterima sistem.
pre-condition blocking dev B
03
Auto-assign meja logic
Logic pemilihan meja otomatis berdasarkan kapasitas tamu dan ketersediaan. Bagian dari proses createReservation().
core logic blocking dev B
🔗 Interface Contract — wajib di shared types
createReservation() → reservationId
Dev A harus expose fungsi ini dengan return type yang jelas. Dev B akan mengonsumsi reservationId untuk membuat payment. Tanpa ini, Dev B tidak tahu ke mana harus link payment-nya.
// shared/types/reservation.ts
export async function createReservation(
  input: ReservationInput
): Promise<{ reservationId: string }>

// Dev B pakai ini:
const { reservationId } = await createReservation(input)
await createPayment({ reservationId, amount, type })
setelah fase 1 selesai → unlock
B
Dev B — Payment Integration API
Midtrans Snap · Webhook · Refund · Payment status
✓ bisa mulai
Fase 2 — Event Contract untuk Dev C
Dev C butuh event yang di-emit Dev A untuk setup notification triggers
04
Emit event: reservasi_created
Setiap kali reservasi berhasil dibuat, Dev A wajib emit event ini. Dev C akan subscribe dan trigger notifikasi ke admin dan tamu.
event contract blocking dev C
05
Emit event: reservasi_cancelled
Saat tamu cancel via token, Dev A emit event ini. Dev C subscribe untuk kirim notifikasi pembatalan ke admin.
event contract blocking dev C
setelah event contract ada → unlock
C
Dev C — Check-in + Notification API
Push VAPID · Subscribe events · Notif triggers · Read status
✓ bisa mulai
Fase 3 — Tidak blocking, urutan bebas
Dikerjakan setelah Fase 1 & 2, tidak memblok siapapun
06
Generate & validate cancel_token
Token unik yang dikirim ke tamu via email/WA agar bisa cancel reservasi tanpa login ke sistem.
independent
07
Cancel reservasi via token endpoint
Endpoint publik yang menerima cancel_token dan memproses pembatalan. Akan trigger emit event reservasi_cancelled.
independent
08
Admin filter & update status reservasi API
Endpoint untuk panel admin: list reservasi dengan filter (tanggal, status, sesi) dan update manual status (confirmed, no_show, dll).
admin only
